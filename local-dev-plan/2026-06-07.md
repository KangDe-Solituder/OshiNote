# 2026-06-07 Development Log

## Project Purpose

OshiNote is a local-first desktop memory workspace for VTuber, livestream, ASMR, and oshi culture fans. It is built for a user who wants a quiet private place to keep long-lived emotional records: stream impressions, fragments worth saving, translations, images, tags, and scrapbook-like journal pages centered around each oshi.

The goal is not to become a generic productivity app. OshiNote should feel like a personal archive and creative desk:

- Private by default: core data lives in a local SQLite database through Tauri.
- Oshi-centered: every major workflow can be scoped to one oshi, while global library views still exist for maintenance.
- Gentle and expressive: themes, motion, journal covers, page stickers, and illustration layouts should make record keeping feel warm instead of administrative.
- Useful over time: notes, tags, search, archives, illustrations, and journal pages should make old memories easy to rediscover.
- AI-assisted but user-led: translation and writing refinement help polish text, but the user's memory and taste remain the source of truth.

## Experience Principles

The product should open directly into usable surfaces, not marketing-style landing pages. Screens should feel calm, legible, and tactile:

- Navigation is module-based: oshis, notes, illustrations, tags, export, settings, and per-oshi spaces.
- Repeated work should be efficient: lists, filters, compact cards, quick actions, and predictable back behavior matter more than decorative chrome.
- Creative work should be constrained first, expandable later: Journal starts with a manageable canvas and grows only when content needs more room.
- Names should match the user's mental model. In Journal, `Archive` is the collection/folder-like object, while `Loose page` is a standalone single layout that can later be collected into an archive.
- Motion should be consistent with the global UI motion setting. Page switches, popovers, side panels, and gallery transitions should all use shared theme motion helpers.

## Current Feature Surface

### Global Shell

- `src/App.tsx` defines React Router routes for home, oshis, per-oshi modules, global notes, global illustrations, tags, export, and settings.
- `src/components/layout/MainLayout.tsx` and `Sidebar.tsx` provide the desktop shell and module navigation.
- `OshiModuleHeader.tsx` provides consistent per-oshi headers with back navigation, icons, subtitles, and optional actions.
- Theme and motion state are managed through `themeStore` and theme CSS files under `src/styles/themes/`.

### Oshi Management

- Users can create, edit, and view oshis as primary content owners.
- Oshi records include identity, theme color, avatar, description, and activity links.
- `OshiOverviewPage.tsx` summarizes notes, illustrations, journal archives, tags, recent notes, recent illustrations, and recent journal archives for one oshi.
- `features/oshis` contains the oshi and archive service layer used by overview and editor flows.

### Notes

- Notes can be created globally or under an oshi/archive context.
- Note data supports title, rich content JSON, plain text, tags, favorites, source URL, optional oshi ownership, and optional archive ownership.
- `NotesPage.tsx` provides global maintenance: query, oshi filter, archive filter, tag filter, ownership filter, favorite filter, sorting, and pagination.
- `NoteEditorPage.tsx` hosts editing workflows and metadata assignment.
- Search is backed by SQLite FTS through `notes_fts`, rebuilt during database initialization when needed.

### Tags

- Tags are stored on notes as serialized arrays and surfaced through global tag pages and per-oshi tag pages.
- Tag pages support browsing, filtering, and navigating back to matching notes.
- Tag graph and tag detail components are designed to make recurring topics discoverable over time.

### Illustrations

- Illustration library support has been added for official art, fanart, screenshots, and images that may later be used in journal layouts.
- `IllustrationsPage.tsx` is the global library surface.
- `OshiIllustrationsPage.tsx` is the per-oshi illustration surface.
- `features/illustrations/illustrationService.ts` handles illustration records.
- `services/media/illustrationMedia.ts` resolves local media paths for previews and thumbnails.
- Journal items can now reference either a note or an illustration.

### Journal

Journal is the expressive archive surface. It is designed around two visible object types:

- `Archive`: a folder/book-like collection of pages.
- `Loose page`: a standalone page, formerly treated as a postcard, that can remain independent or be collected into an archive.

Implemented Journal behavior:

- Journal home shows archives and loose pages together in a single surface.
- `New` is one menu with choices for `Archive` and `Loose page`.
- Archives have cover color, cover style, decoration, date label, and page count.
- Loose pages have page metadata and can be collected into an archive.
- Opening an archive first shows a Pages gallery instead of jumping directly into an editor page.
- Opening a page enters the page editor.
- Back behavior is two-level inside an archive: page editor -> Pages gallery -> Journal home.
- `Pages` can be opened as a side panel from the editor.
- Page gallery and side panel thumbnails are generated from real journal items rather than decorative placeholders.
- Page editor supports placing notes and illustrations, page setup, auto arrange, deleting pages, and saving a page as a loose page.
- Page canvas starts at a constrained minimum size and expands to contain placed items, up to a maximum size.
- Moving items back into the minimum bounds naturally shrinks the canvas back down.
- Canvas zoom controls support overview work without changing stored item coordinates.

### AI Tools

- AI provider abstraction supports OpenAI, Claude, Gemini, and local OpenAI-compatible providers.
- Translation and refinement panels use provider configuration from local settings.
- AI features are intentionally helper tools, not automatic data owners.

### Export

- Export module exists as a separate route and service namespace.
- Current design direction is to support JSON, Markdown, and text output while keeping local data ownership clear.

## Frontend Design

### Theme System

Themes are CSS variable driven. Each theme defines:

- background surfaces: `--color-bg-primary`, `--color-bg-secondary`, `--color-bg-tertiary`, `--color-bg-card`
- text colors: `--color-text-primary`, `--color-text-secondary`, `--color-text-muted`
- accent colors: `--color-accent`, `--color-accent-hover`, `--color-accent-soft`
- borders and glass panel variables
- font family and border radius scale

Journal-specific colors are derived from global theme variables in `src/index.css` with `color-mix(...)`, so stickers, canvas dots, and page previews stay visually aligned with the selected theme.

### Motion System

Motion is centralized in `src/components/features/themes/uiMotion.ts`:

- `useUiMotionSeconds()` reads the global motion duration.
- `usePageTransition()` is used for page/gallery/canvas changes.
- `usePanelTransition()` is used for expanding settings panels.
- `usePopoverTransition()` is used for menus and popovers.

The intent is that every interaction respects user-configured motion speed, including `off`, `fast`, `normal`, and `slow`.

### UI Composition

- Buttons use shared UI primitives from `components/ui`.
- Icons come from Lucide React.
- Dense toolbars are preferred over explanatory panels once a module is understandable.
- Cards are used for repeated items, not nested page sections.
- Journal toolbar actions are consolidated: `Add` groups note/illustration/page creation; more actions holds auto arrange and loose-page conversion.

## Backend And State Design

### Runtime

The app runs as a Tauri v2 desktop application with a React/Vite frontend and Rust/Tauri shell. SQLite access uses `@tauri-apps/plugin-sql`.

### Database Initialization

`src/database/index.ts` owns database loading and defensive schema readiness:

- `getDb()` memoizes the database connection.
- `initializeDb()` runs recovery, migrations, and schema ensure steps.
- `runMigrations()` applies static migrations, then compatibility ensures for evolving schemas.
- Recovery helpers repair interrupted note ownership migrations.

This project is still evolving quickly, so several `ensure...Schema` functions exist to keep older local databases compatible without asking users to manually reset data.

### Stores

Zustand stores are used for frontend state:

- `journalStore` owns archive/page/item state, loading, active IDs, unplaced notes, unplaced illustrations, and journal mutations.
- `noteStore` owns note lists, pagination, filters, note updates, and favorites.
- `themeStore` owns selected theme and UI motion duration.
- Other stores cover oshis, archives, AI, editor, and sidebar behavior.

### Services

Service modules isolate database logic from views:

- `journalService.ts` handles journal archives, pages, loose pages, journal items, note placement, illustration placement, collecting loose pages, detaching pages, and item layout/style updates.
- `noteService.ts` handles note CRUD and query workflows.
- `illustrationService.ts` handles illustration records.
- `oshiService.ts` and `archiveService.ts` handle oshi and note archive data.

The views should call stores where possible, and stores should call services. Direct service calls from a component are acceptable for read-only auxiliary data, such as Journal page preview item loading, but should remain scoped.

## Database Design

### Core Tables

- `oshis`: primary identity records.
- `archives`: note organization archives under oshis.
- `notes`: note content, tags, favorites, source URLs, optional oshi/archive ownership.
- `notes_fts`: full-text index over note title, plain text, and tags.
- `settings`: local app settings and migration markers.

### Illustration Tables

- `illustrations`: image metadata, ownership, file paths, dimensions, source/artist information, tags, favorite state, and archive state.

### Journal Tables

- `journal_books`: visible `Archive` records for Journal. Fields include oshi ID, title, description, cover style, cover color, cover decoration, date label, sort order, and timestamps.
- `journal_pages`: pages inside archives or standalone loose pages. Fields include nullable book ID, oshi ID, page type, title, description, date label, standalone flag, page index, background, and timestamps.
- `journal_items`: placed objects on pages. Items can reference a note or an illustration. Layout fields include x, y, width, height, rotation, z-index, sticker style, color, border style, and timestamps.

Journal intentionally keeps layout item positions in database records, while canvas size is derived at runtime from placed items. This avoids another persisted configuration surface and lets pages shrink back naturally when content moves inward.

## Today's Journal Update

The main work today was turning the afternoon Journal update into a clearer product model and more coherent editing workflow.

### Naming And Information Architecture

Removed the visible split between `Books` and `Postcards`. The new language is:

- `Archive`: collection of pages.
- `Loose page`: standalone page that can later be collected into an archive.
- `Journal`: the single top-level module name.

The intermediate `Archive Desk` heading was removed because it created a confusing second product layer under `Journal`.

### Journal Home

- Journal home now shows archives and loose pages together.
- Search, filter, new menu, and view toggle are in one compact toolbar.
- Empty state says the journal is waiting, not the archive desk.
- Recent overview labels were updated from journal books to journal archives.

### Archive And Page Navigation

- Opening an archive shows a Pages gallery first.
- Clicking a page opens the editor.
- Back from the editor returns to the Pages gallery.
- Back from the Pages gallery returns to Journal home.
- This matches the user's mental model: archive -> page collection -> concrete page.

### Page Thumbnails

The previous thumbnails were decorative placeholders, so they did not match actual page contents. The new implementation:

- Loads `journal_items` for each page.
- Computes each page preview from real item layout data.
- Draws lightweight note/illustration blocks at scaled x/y/width/height positions.
- Preserves page background style and item rotation.

This makes page selection practical because thumbnails now communicate where content actually lives.

### Page Editor

- Toolbar reduced visual noise.
- `Add` menu groups note, illustration, and page creation.
- More menu contains auto arrange and saving a page as a loose page.
- `Page setup` handles title, date label, description, and background.
- `Pages` side panel is expandable/collapsible and uses the same real thumbnails.
- Note and illustration placement share the same journal item model.

### Canvas Behavior

The editor canvas now starts at the minimum page size:

- Minimum: `980 x 700`.
- Maximum: `2200 x 1600`.
- The canvas expands when placed items move beyond the current bounds.
- The canvas shrinks back when items move inside the minimum bounds.
- Stored item coordinates remain unchanged; canvas size is derived from the item bounding box.
- Zoom controls allow `50%` to `125%` viewing without changing layout data.
- Dragging accounts for zoom so pointer movement maps correctly to stored coordinates.

### Animation

Journal page/gallery transitions, side panel expansion, page setup panel, and popover menus now use the shared motion helpers. This keeps Journal aligned with the global UI motion setting.
